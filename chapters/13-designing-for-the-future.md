# Chapter 13: Designing for the Future

> "Harness engineering contains context engineering, which contains prompt engineering. A good harness still requires good context, and good context still requires good prompts."

## 13.1 The Harness Engineering Era

By 2026, the field has recognized that context engineering is itself a component of a larger system design problem. The term "harness engineering" describes the full stack of infrastructure surrounding an agent: the loop that orchestrates inference calls, the sandbox that provides tool execution, the persistence layer that maintains state, the evaluation system that measures quality, and the compaction system that manages context.

OpenAI's experience building a fully agent-generated codebase (described in *Harness Engineering*, February 2026) crystallizes this:

> "The first commit to an empty repository landed in late August 2025. The initial scaffold—repository structure, CI configuration, formatting rules, package manager setup, and application framework—was generated by Codex CLI using GPT-5."

Their key insight: **agents are most effective in environments with strict boundaries and predictable structure.** They built the application around a rigid architectural model with:
- Fixed layers per business domain (Types → Config → Repo → Service → Runtime → UI)
- Strictly validated dependency directions
- Mechanical enforcement via custom linters
- Structural tests that verify architectural compliance

This is the opposite of "let the agent figure it out." It is "define the constraints precisely, then give the agent maximum autonomy within those constraints."

## 13.2 Anthropic's Three-Agent Architecture

Anthropic's March 2026 paper on harness design for long-running applications describes a three-agent architecture that addresses both context management and quality assurance:

### The Planner

Decomposes the overall task into tractable chunks. Produces a structured plan that serves as the coordination artifact.

### The Generator

Executes each chunk of the plan. Works within a focused context for the current subtask. Outputs artifacts (code, documents, configurations) that can be independently evaluated.

### The Evaluator

Reviews the generator's output against specific criteria. Provides structured feedback. The critical innovation: evaluation criteria are decomposed into concrete, gradable terms rather than subjective judgments.

**Key finding**: This architecture produced a 39% improvement in frontend design quality compared to single-pass generation. The evaluator's feedback loop drives the generator toward stronger outputs across iterations.

### Context Resets as a Feature

The three-agent architecture uses context resets between agents. Each agent starts with a fresh context loaded from:
- The original task description
- The current plan state
- Previous artifacts (code, documents)
- Structured feedback from the evaluator

This eliminates both context pollution and context anxiety. Each agent focuses on its current job with a clean, purpose-built context.

## 13.3 Principles for Future-Proof Design

Drawing across all the systems, research, and production lessons in this book, several principles emerge for designing context management systems that will remain relevant as models improve:

### 1. Design for the Summary, Not the Conversation

Every piece of information the agent needs to carry across a compaction event should be explicitly represented in the conversation—the summarizer needs material to work with. Better yet: write important state to files, which survive compaction entirely.

### 2. Make Context Management Model-Agnostic

Manus's approach—betting on context engineering rather than training custom models—proved prescient:

> "If model progress is the rising tide, we want Manus to be the boat, not the pillar stuck to the seabed."

Context management strategies should be orthogonal to the underlying model. A well-designed harness should work with GPT-5, Claude Opus 4.6, Gemini 2.5 Pro, or any future model without fundamental redesign. The models get better; the context management gets better independently.

### 3. Enforce Invariants, Not Implementations

OpenAI's approach to agent-generated code: enforce boundaries centrally, allow autonomy locally. Define *what* the architecture requires (data validation at boundaries, fixed dependency directions) and let the agent decide *how* to implement it (the agent chose Zod, but the harness didn't specify Zod).

This principle applies to context management too: define the budget, the compaction policy, and the memory architecture. Don't prescribe the exact content of every context component.

### 4. Build for Rhythmic Operation

Agents don't run forever in one session. They pulse: work, compact, continue. Or: work, reset, hand off. Design state management for this rhythm:

- State files that any session can read and resume from
- Handoff artifacts that carry context across agent boundaries
- Memory systems that accumulate experience across sessions

Assume the agent *will* lose its context window. Make recovery cheap.

### 5. Measure Everything

The most effective teams treat context management as infrastructure with metrics:

| Metric | What It Measures | Target |
|--------|-----------------|--------|
| KV-cache hit rate | Prefix stability | >70–80% |
| Tokens per task completion | Context efficiency | Trending down |
| Compaction frequency | Context pressure | Stable or decreasing |
| Task completion rate | End-to-end quality | >90% |
| Cost per task | Economic viability | Trending down |
| Context utilization % | Budget management | 40–70% average |

Without measurement, optimization is guesswork.

### 6. Keep it Simple

Anthropic's recurring advice deserves to be the final word:

> "Do the simplest thing that works."

Models are getting smarter. The patterns that require elaborate engineering today may become unnecessary as models develop better native context handling. Start with the simplest solution (prompt engineering), add complexity only when measurement shows it's needed (context editing → compaction → multi-agent), and remain willing to simplify when better models arrive.

## 13.4 Open Research Questions

Several fundamental challenges remain unsolved as of early 2026:

### Continual Consolidation

How should an agent consolidate thousands of episodic memories into general knowledge? Current systems store individual memories but don't effectively abstract from them. Humans do this naturally (individual interactions become general skills), but no agent memory system reliably reproduces it.

### Causally Grounded Retrieval

Current retrieval systems find semantically similar memories, but agents need memories that are *causally* relevant. "I encountered a similar error" is less useful than "I encountered this exact error because of this exact cause." Semantic similarity doesn't capture causal structure.

### Trustworthy Reflection

Reflection (the agent generating lessons from its experience) can produce incorrect generalizations. An agent that "learns" the wrong lesson from a debugging session may consistently misdiagnose similar issues in the future. Quality control for agent-generated reflections is an open problem.

### Learned Forgetting

Which memories should be forgotten? Outdated API patterns, deprecated configurations, superseded architectural decisions—these should be pruned, but determining what's outdated requires understanding the current state of the codebase, which itself requires context.

### Multimodal Memory

Current memory systems are primarily text-based. As agents increasingly interact with visual interfaces (web pages, design tools, dashboards), memory systems will need to handle screenshots, diagrams, and visual state alongside textual information.

## 13.5 The Trajectory of the Field

Looking at the progression from 2022 to 2026:

```
2022: "Write a good prompt"
2023: "Add few-shot examples and chain-of-thought"
2024: "Use RAG and tool calling"
2025: "Engineer the full context: system prompt, tools, history, retrieval"
2026: "Design the full harness: context + execution + persistence + evaluation"
```

The trend is clear: the unit of design is expanding. From a single prompt to a context configuration to a full system architecture. Each layer contains the previous ones—you still need good prompts and good context within a good harness.

But a counter-trend is also visible: **better models require less engineering.** Anthropic found that Opus 4.6's capabilities allowed them to drop multi-session handoff complexity, relying on compaction alone. As models improve at native context handling, some of today's elaborate engineering will become unnecessary.

The optimal strategy is to build modular, layered systems where each component can be simplified or removed as models improve. Don't build a cathedral when the ground is still shifting. Build well-designed modules that can be rearranged.

## 13.6 Final Thoughts

Context management for long-running LLM agents is, as Manus describes it, "still an emerging science." The patterns in this book represent the state of the art as of early 2026—hard-won through production deployment at the world's leading AI companies and validated across millions of agent sessions.

The core insight is simple: **the model's intelligence is not the bottleneck. The context is.** Every technique in this book—compaction, retrieval, external memory, multi-agent isolation, cache optimization, experience accumulation, dynamic discovery—serves a single goal: ensuring the model sees the right information at the right time.

For those building agents that run hard tasks over long horizons, context engineering is not an optimization. It is the engineering discipline that determines whether your agent completes its task or degrades into incoherence. Master it, and your agents will be limited only by the frontier of model capability. Neglect it, and even the most powerful model will fail.

The agentic future will be built one context at a time. Engineer it well.
