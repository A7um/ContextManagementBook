## 13.1 The Harness Engineering Era

By 2026, the field has recognized that context engineering is itself a component of a larger system design problem. The term **harness engineering** describes the full stack of infrastructure surrounding an agent: the loop that orchestrates inference calls, the sandbox that provides tool execution, the persistence layer that maintains state, the evaluation system that measures quality, and the compaction system that manages context.

OpenAI's experience building a fully agent-generated codebase (described in *Harness Engineering*, February 2026) crystallizes this:

> "The first commit to an empty repository landed in late August 2025. The initial scaffold—repository structure, CI configuration, formatting rules, package manager setup, and application framework—was generated by Codex CLI using GPT-5."

The result: a production application built entirely by agents, running in production, serving real users. But the key insight isn't that agents wrote the code—it's *how the harness was designed to make that possible.*

## 13.2 Anthropic's Three-Agent Architecture

Anthropic's March 2026 paper on harness design for long-running applications describes a three-agent architecture that addresses both context management and quality assurance. It produced a **39% improvement in frontend design quality** compared to single-pass generation.

### Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                         PLANNER                              │
│                                                             │
│  Input:  Full task description + constraints                 │
│  Output: Structured plan with discrete chunks               │
│                                                             │
│  "Build a dashboard with:                                    │
│   Chunk 1: Navigation sidebar with 5 menu items             │
│   Chunk 2: Main content area with data table                │
│   Chunk 3: Filter panel with date range + category          │
│   Chunk 4: Chart visualization (bar + line toggle)          │
│   Chunk 5: Export functionality (CSV + PDF)"                │
│                                                             │
│  Each chunk specifies:                                       │
│  - Inputs (what files/data to work from)                    │
│  - Outputs (what files to produce)                          │
│  - Acceptance criteria (concrete, testable conditions)       │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                        GENERATOR                             │
│                                                             │
│  Input:  Current chunk from plan + relevant files            │
│  Output: Code artifacts + screenshots                        │
│                                                             │
│  Context is RESET between chunks:                            │
│  - Fresh context loaded from plan + chunk spec              │
│  - Previous artifacts available (read from disk)            │
│  - No conversation history from previous chunks             │
│  - No accumulated context drift                              │
│                                                             │
│  Chunk 3 generator receives:                                │
│  - Plan: "Filter panel with date range + category"          │
│  - Spec: inputs, outputs, acceptance criteria               │
│  - Files: sidebar.tsx, content.tsx (from chunks 1-2)        │
│  - Evaluator feedback (if this is iteration 2+)             │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                        EVALUATOR                             │
│                                                             │
│  Input:  Generator output + acceptance criteria              │
│  Output: PASS/FAIL + structured feedback                     │
│                                                             │
│  Evaluation criteria are CONCRETE, not subjective:          │
│                                                             │
│  ✗ BAD:  "Is the filter panel well-designed?"               │
│  ✗ BAD:  "Does it look good?"                               │
│  ✗ BAD:  "Is the UX intuitive?"                             │
│                                                             │
│  ✓ GOOD: "Does the date range picker support:               │
│           - Preset ranges (7d, 30d, 90d, custom)?           │
│           - Custom date input with validation?               │
│           - Clear button to reset filters?                   │
│           Does the category filter:                          │
│           - Show all available categories?                   │
│           - Support multi-select?                            │
│           - Show count of items per category?                │
│           - Apply filters without page reload?"              │
│                                                             │
│  If FAIL: feedback loops back to Generator (max 3 iters)    │
│  If PASS: move to next chunk                                │
└─────────────────────────────────────────────────────────────┘
```

### Why Context Resets Between Agents

The critical innovation is that each agent starts with a **fresh context** loaded from structured artifacts, not from conversation history:

```python
# Generator context for chunk 3 (iteration 2, after evaluator feedback)
generator_context = {
    "system": GENERATOR_SYSTEM_PROMPT,
    "plan": load_file("plan.md"),              # from Planner
    "chunk_spec": plan.chunks[3],               # current chunk details
    "previous_artifacts": [                     # from disk, not from memory
        "src/components/sidebar.tsx",
        "src/components/content.tsx",
    ],
    "evaluator_feedback": """                   # structured, concrete
        FAIL - 2 criteria not met:
        1. Date range picker missing 'custom' option
        2. Category filter does not show item counts
        
        Specific fix required:
        - Add DateRangePicker custom mode with calendar popup
        - Add count badge next to each category option
    """,
    "user_message": "Implement chunk 3 with the evaluator's feedback."
}
# Total context: ~15K tokens (focused and fresh)
# Compare: single-pass at this point would be ~80K tokens (accumulated)
```

**Benefits of context resets:**
1. **No context drift:** Each generator invocation starts from a clean state
2. **No context anxiety:** The generator never approaches its context limit
3. **Focused attention:** Only the current chunk's context competes for attention
4. **Cheap iteration:** Re-running the generator for a chunk costs ~15K tokens, not ~80K

### When to Use Three-Agent vs. Single-Agent

| Scenario | Architecture | Reasoning |
|----------|-------------|-----------|
| Simple task (<5 min) | Single agent | Overhead of planning/evaluation exceeds benefit |
| Medium task (5-60 min) | Single agent + compaction | Compaction handles context growth adequately |
| Complex task (1-6 hours) | Three-agent | Context resets prevent degradation over many steps |
| Multi-domain task | Three-agent + specialized generators | Each generator can be expert in its domain |

## 13.3 OpenAI's Harness Engineering Principles

OpenAI's harness engineering paper describes specific principles that enabled 6+ hour autonomous Codex runs:

### Rigid Architecture: Fixed Layers

```
┌─────────────────────────────────────────────────┐
│  Layer 6: UI Components                          │
│  Layer 5: Runtime (server startup, middleware)    │
│  Layer 4: Service (business logic)               │
│  Layer 3: Repository (data access)               │
│  Layer 2: Config (environment, feature flags)     │
│  Layer 1: Types (data structures, enums)          │
└─────────────────────────────────────────────────┘

Dependency rules:
  Layer N may depend on Layer N-1 through Layer 1
  Layer N may NEVER depend on Layer N+1 or above
  
  Types has ZERO imports from other layers
  Config imports only from Types
  Repository imports from Types + Config
  Service imports from Types + Config + Repository
  Runtime imports from everything below
  UI imports from everything below
```

### Custom Linters for Enforcement

The linters themselves were generated by Codex:

```python
# Simplified representation of the Codex architectural linter
import ast
import sys
from pathlib import Path

LAYER_ORDER = ["types", "config", "repo", "service", "runtime", "ui"]

ALLOWED_DEPS = {
    "types": set(),
    "config": {"types"},
    "repo": {"types", "config"},
    "service": {"types", "config", "repo"},
    "runtime": {"types", "config", "repo", "service"},
    "ui": {"types", "config", "repo", "service", "runtime"},
}

def get_layer(filepath: str) -> str:
    """Determine which layer a file belongs to based on its path."""
    parts = Path(filepath).parts
    for layer in LAYER_ORDER:
        if layer in parts:
            return layer
    return "unknown"

def check_imports(filepath: str) -> list[str]:
    """Check that all imports respect layer boundaries."""
    violations = []
    source_layer = get_layer(filepath)
    
    if source_layer == "unknown":
        return violations
    
    with open(filepath) as f:
        tree = ast.parse(f.read())
    
    for node in ast.walk(tree):
        if isinstance(node, (ast.Import, ast.ImportFrom)):
            module = node.module if isinstance(node, ast.ImportFrom) else node.names[0].name
            if module:
                target_layer = get_layer(module.replace(".", "/"))
                if target_layer != "unknown" and target_layer not in ALLOWED_DEPS[source_layer]:
                    violations.append(
                        f"{filepath}:{node.lineno}: Layer violation: "
                        f"'{source_layer}' cannot import from '{target_layer}'"
                    )
    
    return violations
```

### The Principle: "Enforce Invariants, Not Implementations"

```
INVARIANTS (enforced mechanically):
  ✓ "Data validation happens at every layer boundary"
  ✓ "Dependency direction: Types → Config → Repo → Service → Runtime → UI"
  ✓ "Every public function has a type signature"
  ✓ "No direct database access outside the Repo layer"

IMPLEMENTATIONS (agent decides):
  ✗ "Use Zod for validation" (agent chose this, but it wasn't prescribed)
  ✗ "Use Prisma for database access" (agent chose this)
  ✗ "Use React Query for data fetching" (agent chose this)
  ✗ "Use shadcn/ui for components" (agent chose this)
```

The harness defines *what* must be true (invariants) and verifies it mechanically (linters, structural tests, CI checks). The agent decides *how* to satisfy those invariants. This gives maximum autonomy within precisely defined boundaries—exactly what enables 6+ hour runs without human supervision.

## 13.4 Five Design Principles for Future-Proof Systems

Drawing across all the systems, research, and production lessons in this book, five principles emerge for designing context management systems that remain relevant as models improve.

### Principle 1: Design for the Summary, Not the Conversation

Every piece of information the agent needs to carry across a compaction event should be explicitly represented where a summarizer can find it. Better yet: write it to files, which survive compaction entirely.

```python
# BAD: Critical state only exists in conversation context
def handle_migration_step(state, step_result):
    # The fact that we're on step 3 of 7 is only in conversation
    # After compaction, this information may be lost or degraded
    return f"Step {state['current_step']} complete. Result: {step_result}"

# GOOD: Critical state written to file, conversation references it
def handle_migration_step(state, step_result):
    state["current_step"] += 1
    state["completed_steps"].append({
        "step": state["current_step"],
        "result": step_result,
        "timestamp": datetime.now().isoformat()
    })
    
    # Write state to file — survives compaction
    with open("migration_state.json", "w") as f:
        json.dump(state, f, indent=2)
    
    return (
        f"Step {state['current_step']}/7 complete. "
        f"State saved to migration_state.json. "
        f"Next: {state['remaining_steps'][0]}"
    )
```

**The test:** If you compacted the conversation right now, would the agent know:
- What task it's working on?
- What it's already completed?
- What's left to do?
- What decisions it made and why?

If any of these would be lost, you're designing for the conversation, not for the summary.

### Principle 2: Make Context Management Model-Agnostic

Manus's approach—betting on context engineering rather than training custom models—proved prescient:

> "If model progress is the rising tide, we want Manus to be the boat, not the pillar stuck to the seabed."

```python
# BAD: Model-specific context management
class ContextManager:
    def __init__(self, model: str):
        if model == "claude-sonnet-4-6":
            self.compaction_threshold = 167_000  # Anthropic-specific
            self.compaction_method = "anthropic_summary"
            self.cache_strategy = "anthropic_prompt_cache"
        elif model == "gpt-5.3":
            self.compaction_threshold = 180_000  # OpenAI-specific  
            self.compaction_method = "responses_api_compact"
            self.cache_strategy = "openai_prefix_cache"

# GOOD: Model-agnostic with pluggable adapters
class ContextManager:
    def __init__(self, config: ContextConfig):
        self.max_context = config.max_context_tokens
        self.compaction_pct = config.compaction_trigger_pct  # e.g., 0.85
        self.reserve_tokens = config.output_reserve
        
    @property
    def compaction_threshold(self) -> int:
        return int(self.max_context * self.compaction_pct) - self.reserve_tokens
    
    def compact(self, conversation: list[Message]) -> list[Message]:
        """Model-agnostic compaction: summarize + rehydrate recent files."""
        summary = self.summarizer.summarize(conversation)
        recent_files = self.get_recent_files(conversation, n=5)
        return [summary] + recent_files + conversation[-3:]


# Config per model (the only model-specific part)
configs = {
    "claude-sonnet-4-6": ContextConfig(max_context=200_000, compaction_trigger_pct=0.85, output_reserve=20_000),
    "gpt-5.3":           ContextConfig(max_context=256_000, compaction_trigger_pct=0.80, output_reserve=30_000),
    "gemini-2.5-pro":    ContextConfig(max_context=1_000_000, compaction_trigger_pct=0.90, output_reserve=50_000),
}
```

When a new model launches with a 2M context window, you update one config entry. The rest of your system works unchanged.

### Principle 3: Enforce Invariants, Not Implementations

Define *what* the architecture requires and verify it mechanically. Let the agent decide *how.*

```
INVARIANT ENFORCEMENT STACK:

1. Type System (compile-time)
   "Every API response has a typed error union"
   → TypeScript strict mode + discriminated unions
   → Agent can choose the union shape, but it must exist

2. Linters (pre-commit)
   "No cross-layer imports"
   → Custom ESLint rule validates import paths
   → Agent can organize files freely within layers

3. Structural Tests (CI)
   "Every public endpoint has a corresponding test"
   → Script checks test file exists for each route handler
   → Agent can write any test, but one must exist

4. Integration Tests (CI)
   "Auth middleware rejects expired tokens"
   → Test sends expired token, asserts 401
   → Agent can implement auth any way it wants

5. Monitoring (runtime)
   "P99 latency < 200ms for all API endpoints"
   → Alerting on latency regression
   → Agent can use any optimization technique
```

Each layer enforces a what without prescribing a how. The agent has maximum freedom to choose implementations while the system guarantees correctness.

### Principle 4: Build for Rhythmic Operation

Agents don't run forever in one session. They pulse: work, compact, continue. Or: work, reset, hand off. Design for this rhythm:

```python
class AgentState:
    """State that survives any interruption: compaction, session end, crash."""
    
    def __init__(self, state_file: str = "agent_state.json"):
        self.state_file = state_file
        self.state = self._load()
    
    def _load(self) -> dict:
        """Load state from disk. Any new session starts from disk state."""
        try:
            with open(self.state_file) as f:
                return json.load(f)
        except FileNotFoundError:
            return {
                "task": None,
                "plan": [],
                "completed_steps": [],
                "current_step": None,
                "decisions": [],
                "known_issues": [],
                "created_at": datetime.now().isoformat(),
                "last_updated": None
            }
    
    def checkpoint(self):
        """Save current state to disk. Call after every significant action."""
        self.state["last_updated"] = datetime.now().isoformat()
        with open(self.state_file, "w") as f:
            json.dump(self.state, f, indent=2)
    
    def complete_step(self, step: str, result: str):
        self.state["completed_steps"].append({
            "step": step, "result": result,
            "timestamp": datetime.now().isoformat()
        })
        if self.state["plan"]:
            self.state["current_step"] = self.state["plan"][
                len(self.state["completed_steps"])
            ] if len(self.state["completed_steps"]) < len(self.state["plan"]) else None
        self.checkpoint()
    
    def record_decision(self, decision: str, rationale: str):
        self.state["decisions"].append({
            "decision": decision, "rationale": rationale,
            "timestamp": datetime.now().isoformat()
        })
        self.checkpoint()
    
    def resume_prompt(self) -> str:
        """Generate a prompt that lets any agent resume from current state."""
        s = self.state
        completed = "\n".join(f"  ✓ {step['step']}: {step['result']}" 
                              for step in s["completed_steps"])
        remaining = "\n".join(f"  ○ {step}" 
                              for step in s["plan"][len(s["completed_steps"]):])
        decisions = "\n".join(f"  - {d['decision']}: {d['rationale']}" 
                              for d in s["decisions"])
        
        return f"""## Resuming Task: {s['task']}

### Completed
{completed or '  (none yet)'}

### Current Step
  → {s['current_step'] or '(planning)'}

### Remaining
{remaining or '  (all done)'}

### Key Decisions Made
{decisions or '  (none yet)'}

### Known Issues
{chr(10).join('  ⚠ ' + issue for issue in s['known_issues']) or '  (none)'}

Continue from the current step. Read agent_state.json for full details."""
```

**The test:** If the agent crashes right now and a brand-new agent instance starts with only the state file, can it resume the task? If yes, you've designed for rhythmic operation.

### Principle 5: Measure Everything

The most effective teams treat context management as infrastructure with metrics:

```python
class ContextMetrics:
    """Track context management health across sessions."""
    
    def __init__(self):
        self.metrics = []
    
    def record_turn(self, turn_data: dict):
        self.metrics.append({
            "timestamp": datetime.now().isoformat(),
            "kv_cache_hit_rate": turn_data.get("cache_hit_rate", 0),
            "tokens_used": turn_data["prompt_tokens"] + turn_data["completion_tokens"],
            "prompt_tokens": turn_data["prompt_tokens"],
            "completion_tokens": turn_data["completion_tokens"],
            "context_utilization": turn_data["prompt_tokens"] / turn_data["context_window"],
            "tools_loaded": turn_data.get("tools_in_prompt", 0),
            "tools_used": turn_data.get("tools_called", 0),
            "compaction_triggered": turn_data.get("compacted", False),
        })
    
    def report(self) -> str:
        if not self.metrics:
            return "No metrics collected yet."
        
        n = len(self.metrics)
        avg = lambda key: sum(m[key] for m in self.metrics) / n
        
        return f"""# Context Management Metrics

| Metric | Value | Target | Status |
|--------|-------|--------|--------|
| KV-cache hit rate | {avg('kv_cache_hit_rate'):.1%} | >70% | {'✅' if avg('kv_cache_hit_rate') > 0.7 else '❌'} |
| Avg tokens/turn | {avg('tokens_used'):,.0f} | Trending ↓ | {'✅' if self._is_trending_down('tokens_used') else '⚠️'} |
| Context utilization | {avg('context_utilization'):.1%} | 40-70% | {'✅' if 0.4 <= avg('context_utilization') <= 0.7 else '⚠️'} |
| Tool load efficiency | {avg('tools_used')/max(avg('tools_loaded'),1):.1%} | >30% | {'✅' if avg('tools_used')/max(avg('tools_loaded'),1) > 0.3 else '❌'} |
| Compaction rate | {sum(1 for m in self.metrics if m['compaction_triggered'])/n:.1%} | <20% | {'✅' if sum(1 for m in self.metrics if m['compaction_triggered'])/n < 0.2 else '⚠️'} |
| Turns in session | {n} | N/A | — |

### Interpretation
- KV-cache hit rate <70%: Your prompt prefix is unstable. Check for non-deterministic tool ordering.
- Context utilization >70%: You're running too close to limits. Implement dynamic loading.
- Tool load efficiency <30%: Most loaded tools are unused. Implement tool routing.
- Compaction rate >20%: Sessions hit context limits too often. Reduce per-turn context cost.
"""
    
    def _is_trending_down(self, key: str) -> bool:
        if len(self.metrics) < 10:
            return True
        first_half = self.metrics[:len(self.metrics)//2]
        second_half = self.metrics[len(self.metrics)//2:]
        return (sum(m[key] for m in second_half) / len(second_half) <=
                sum(m[key] for m in first_half) / len(first_half))
```

**Targets and what they mean:**

| Metric | Target | Why |
|--------|--------|-----|
| KV-cache hit rate | >70-80% | Below this, you're paying full price for repeated prefix computation |
| Tokens per task | Trending down | Indicates your dynamic loading and compaction are improving |
| Compaction frequency | Stable or decreasing | Frequent compaction means each turn is too expensive |
| Task completion rate | >90% | Below this, context management is degrading task quality |
| Cost per task | Trending down | The north star metric for context engineering ROI |
| Context utilization | 40-70% average | <40% means unused capacity; >70% means constant pressure |

## 13.5 Open Research Questions

Several fundamental challenges remain unsolved as of early 2026:

### Continual Consolidation

How should an agent consolidate thousands of episodic memories into general knowledge? Current systems (ExpRAG, Memori, Brain-of-Markdown) store individual memories and retrieve them. But humans do something more: they abstract from episodes to general skills. An agent that debugged 100 different authentication bugs should develop a general "authentication debugging" skill, not just retrieve the most similar past debug session.

**Promising direction:** Periodic "consolidation passes" where an LLM reviews clusters of related memories and generates principles. Similar to how textbooks are written from lecture notes.

### Causally Grounded Retrieval

Semantic similarity is the basis for all current retrieval systems. But agents need memories that are *causally* relevant, not just *semantically* similar. "I encountered a similar error" is less useful than "I encountered this exact error because the Redis connection pool was exhausted." The causal chain (not just the surface similarity) is what makes a memory actionable.

**Promising direction:** Storing memories as causal graphs (cause → effect → resolution) rather than flat text, then querying by causal pattern matching.

### Trustworthy Reflection

Reflexion and related techniques have the agent generate lessons from experience. But reflections can be wrong. An agent that "learns" from a debugging session that "the database is always the bottleneck" may misdiagnose future issues where the database is fine. Wrong reflections compound: they bias future reasoning, which generates more wrong reflections.

**Promising direction:** Validation loops where reflections are tested against future experiences. Reflections that predict correctly get promoted; those that don't get demoted or deleted. Essentially: unit tests for agent beliefs.

### Learned Forgetting

Which memories should be forgotten? Outdated API patterns, deprecated configurations, superseded architectural decisions—these should be pruned. But determining what's outdated requires understanding the current state of the codebase, which itself requires context. It's a chicken-and-egg problem.

**Current workaround:** Time-based expiry (TTLs on memories) plus explicit invalidation when a contradicting memory is stored. Neither is sufficient.

### Multimodal Memory

Current memory systems are text-only. But agents increasingly interact with visual interfaces. An agent that debugged a CSS layout issue by looking at a screenshot should be able to store and retrieve that visual context. A design review that produced annotated mockups should be retrievable when similar design work comes up.

**Promising direction:** Vision-language models for memory encoding—store a (screenshot, description) pair and retrieve on either visual or textual similarity.

## 13.6 The Trajectory of the Field

```
2022  ──►  "Write a good prompt"
           Single turn. No tools. No memory. Success = well-crafted prompt.

2023  ──►  "Add few-shot examples and chain-of-thought"  
           Still single turn, but structured. ReAct, Tree-of-Thought.

2024  ──►  "Use RAG and tool calling"
           Multi-turn. External knowledge. Tool use. But context is static.

2025  ──►  "Engineer the full context"
           Dynamic loading. Compaction. KV-cache optimization. 
           Context engineering as a named discipline.

2026  ──►  "Design the full harness"
           Context + execution + persistence + evaluation + enforcement.
           Harness engineering. Systems that run for hours autonomously.

2027? ──►  "The model manages its own context"
           Models with native memory management, learned compaction,
           and self-directed retrieval. Less engineering, more capability.
```

The trend is clear: the unit of design is expanding. From a single prompt to a context configuration to a full system architecture. Each layer contains the previous ones—you still need good prompts within good context within a good harness.

### The Counter-Trend: Better Models Need Less Engineering

A counter-trend is equally visible: **better models require less engineering.** Specific examples:

- **Anthropic found that Opus 4.6's capabilities allowed them to drop multi-session handoff complexity**, relying on compaction alone where previously they needed explicit state transfer between sessions.
- **Context anxiety reduction with larger windows** (Devin's 1M window trick) means models self-manage context pressure better when given room.
- **Built-in context awareness** (Claude Sonnet 4.6) means the model makes intelligent decisions about when to compact, when to write to files, and when to delegate—reducing the need for external trigger logic.
- **Tool search** (Anthropic's `defer_loading`) means the model discovers tools itself, reducing the need for manual tool routing logic.

The implication: some of today's context engineering will become unnecessary as models improve. The system-prompt-as-table-of-contents pattern may be replaced by models that naturally manage their own information loading. The compaction triggers may be replaced by models that proactively compact when they sense degradation.

### What Won't Go Away

Even as models improve, certain aspects of context management will remain essential:

1. **External persistence.** Models will always need files, databases, and other external stores for state that must survive beyond a session.
2. **Enforcement.** Architectural constraints, linters, and structural tests don't become less necessary with better models—they become more important as agents take on larger tasks.
3. **Evaluation.** The Planner-Generator-Evaluator pattern addresses a fundamental problem (quality assurance) that better models don't eliminate.
4. **Measurement.** You can't optimize what you don't measure, regardless of model capability.

## 13.7 The Final Principle: Simplicity

Anthropic's recurring advice:

> "Do the simplest thing that works."

The progression:

```
Step 1: Can you solve this with a better prompt?
        → Yes: stop here. You're done.
        
Step 2: Can you solve this with prompt + simple context assembly?
        → Yes: add system prompt, few-shot examples, RAG. Stop.
        
Step 3: Can you solve this with compaction?
        → Yes: add auto-compaction at 85% threshold. Stop.
        
Step 4: Can you solve this with context editing?
        → Yes: add clear_tool_uses + thinking block cleanup. Stop.
        
Step 5: Can you solve this with dynamic loading?
        → Yes: add progressive disclosure for tools/skills/memory. Stop.
        
Step 6: Can you solve this with multi-agent?
        → Yes: add Planner-Generator-Evaluator or subagent delegation. Stop.
        
Step 7: Can you solve this with custom enforcement?
        → Yes: add linters, structural tests, architectural constraints. Stop.
        
Step 8: You need a full harness.
        → Build it modularly so you can remove layers when better models arrive.
```

Most teams jump to Step 6 or 7 when Step 2 or 3 would suffice. Every layer of complexity adds failure modes, maintenance burden, and cognitive overhead. The best systems are the simplest ones that reliably complete their tasks.

Build modular layers. Measure each layer's contribution. Remove layers that better models make unnecessary. This is how you build systems that last—not by building cathedrals on shifting ground, but by assembling well-designed modules that can be rearranged as the landscape changes.

## 13.8 Key Takeaways

1. **The three-agent architecture (Planner → Generator → Evaluator) is the state-of-the-art for complex tasks.** 39% quality improvement from context resets + concrete evaluation criteria. Use it for tasks longer than 1 hour.

2. **Enforce invariants, not implementations.** Define architectural constraints mechanically (linters, structural tests, CI). Give agents maximum autonomy within those constraints. This is what enables 6+ hour autonomous runs.

3. **Design for the summary.** If you compacted the conversation right now, would the agent know what to do? Write critical state to files. Reference files in conversation.

4. **Keep context management model-agnostic.** Use config objects for model-specific parameters. The rest of your system should work with any model.

5. **Build for rhythmic operation.** Assume the agent will lose its context. Make recovery cheap: state files, resume prompts, checkpoint-on-every-action.

6. **Measure KV-cache hit rate, tokens per task, compaction frequency, and task completion rate.** Without metrics, optimization is guesswork.

7. **Start simple, add complexity only when measurement shows it's needed.** Most agents need a good prompt + compaction + file-based state. Few need multi-agent + custom enforcement + dynamic loading.

8. **Better models will simplify context engineering, but won't eliminate it.** External persistence, enforcement, evaluation, and measurement are durable needs regardless of model capability.

9. **The field is moving from context engineering to harness engineering.** Context is a component of the harness. A good harness still requires good context, and good context still requires good prompts. Build each layer well.
